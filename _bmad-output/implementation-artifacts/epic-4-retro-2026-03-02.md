## 2. Ce qui a bien fonctionné

### Points techniques

- **DraftTrancheService : design extensible** — Le service a été étendu deux fois (4.2 : validation tranche, 4.5 : exclusivité) sans ajouter de nouvelle dépendance, en utilisant `DraftPickRepositoryPort` déjà injecté. Excellent exemple d'OCP.
- **Architecture guards omniprésents** : `NamingConventionTest` a forcé `DraftPickOrchestrator` → `DraftPickOrchestratorService` et `DraftTrancheValidator` → `DraftTrancheService`. Ces renommages auraient été oubliés sans les tests d'architecture automatisés.
- **@Bean Random injectable** : La décision d'injecter `Random` comme bean Spring (Story 4.1) permet des tests déterministes sur le coin flip (4.4) et le shuffle des participants (4.1). Pattern "test seam" bien appliqué.
- **Réutilisation infrastructure** : `findPickedPlayerIdsByDraftId(UUID)` (ajouté en 4.2) a été réutilisé en 4.5 et 4.6 sans modification. Une seule méthode UUID-based sert trois cas d'usage.
- **`IllegalStateException` → 409 CONFLICT** : Story 4.3 a ajouté ce handler en 5 lignes, résolvant les erreurs HTTP 500 qui auraient été visibles en production sur toutes les violations de fenêtre simultanée.
- **Stories 4.3/4.4 "gap-filling"** : Bien que la majorité du code ait été implémenté dans JIRA-UX-003c, les stories BMAD ont identifié et comblé les gaps (handler exception manquant, tests controller manquants) de façon structurée.

### Points process

- **Code reviews sur 100% des stories** : Toutes les 6 stories de l'epic ont été reviewées — c'est le premier epic avec 100% de couverture code review.
- **6 MEDIUM résolus** : Aucun finding MEDIUM n'est resté ouvert. Les findings MEDIUM sont systématiquement fixés avant merge depuis Epic 2.
- **Findings de sécurité** : 3 stories ont nécessité la création de tests de sécurité (4.1, 4.3, 4.6) — tous créés en code review. Le pattern "controller sans test d'autorisation" est détecté systématiquement maintenant.

## 3. Ce qui peut être amélioré

### Gaps identifiés

- **`IllegalStateException`→409 trop large (M1 non fixé de 4.3)** : Le handler `@ExceptionHandler(IllegalStateException.class)` dans `DomainExceptionHandler` intercepte **toutes** les `IllegalStateException` de l'application — Spring internals, Hibernate, toute assertion. Si Spring lève une ISE interne, elle sera retournée comme `409 DRAFT_WINDOW_VIOLATION` au client. C'est un bug potentiel silencieux. La solution est de créer `DraftWindowViolationException` et remplacer les `throw new IllegalStateException(...)` dans `DraftSimultaneousService`.
- **O(n) `List.contains()` — deux fois** : `requirePlayerNotAlreadyPickedInDraft()` dans `DraftTrancheService` (4.5) et `AdminDraftRosterService` (4.6) utilisent tous deux `List.contains()` — O(n). Le code review a identifié le problème deux fois (L1 dans 4.5 et L2 dans 4.6) sans être fixé. Les deux méthodes devraient utiliser `new HashSet<>(ids).contains(playerId)`.
- **16 findings LOW accumulés** : Avec ~16 LOW en attente après cet epic, la dette LOW dépasse le seuil de 8 défini dans la rétro Epic 3. Une story "dette technique" est due.
- **Pas de test d'autorisation ADMIN dans le template** : 3 stories consécutives (4.1, 4.3, 4.6) ont nécessité la création de tests de sécurité en code review. Ce pattern récurrent indique que le template de story ne demande pas explicitement "tests d'autorisation Spring Security" pour chaque controller.
- **Stories 4.3/4.4 quasi-vides** : Les stories correspondent à de la validation de code déjà implémenté (JIRA-UX-003c). Le process BMAD prévoit des stories d'implémentation — quand le code existe déjà, le story scope est réduit à la documentation des gaps. C'est fonctionnel mais laisse une impression de "stories incomplètes" dans le backlog.
- **`pickFirstConflict().orElse(null)` (M1 de 4.4 non fixé)** : Anti-pattern null-propagation. Retourner `Optional<ConflictResolutionResponse>` est plus sûr.

## 4. Élicitation Avancée — 5 Méthodes

### 4.1 Hindsight Reflection — Vue depuis mars 2026 (2 epics plus tard)

**Pattern 1 — `DraftTrancheService` comme point d'extension sans new dep**
En 4.2, le service a été créé avec 5 deps. En 4.5 et 4.6, d'autres services ont réutilisé `findPickedPlayerIdsByDraftId` directement. En Epic 5 (swaps/trades), le service n'a pas été touché. L'architecture a bien tenu.

**Pattern 2 — `IllegalStateException`→409 : le risque latent**
Après Epic 4, ce handler trop large est resté en production. En Epic 5 et 6, aucune ISE inattendue n'a été retournée au client (par chance ou parce que Spring n'en émet pas sur ce code path). Le risque est réel mais ne s'est pas matérialisé dans ce sprint.

**Pattern 3 — JIRA tickets vs BMAD stories : la frontière floue**
JIRA-UX-003c a implémenté `DraftSimultaneousService` et `DraftAsyncWindow` avant que les stories BMAD 4.3 et 4.4 ne soient créées. Ce pattern "code avant story" crée des stories quasi-vides mais est pragmatique quand le développement UX s'accélère. La solution : créer les stories BMAD avant le code, ou documenter explicitement le code pré-existant en "Technical Context Already Done".

**Pattern 4 — Tests de sécurité : le 4ème pattern récurrent**
Stories 3.2, 3.3, 3.4, 4.1, 4.3, 4.6 ont toutes nécessité des tests de sécurité créés en code review. 6 fois le même finding. La cause racine est claire : le template de story ne demande pas de tests d'autorisation.

### 4.2 Lessons Learned Extraction — 5 Leçons

#### Leçon L1 — Jamais `IllegalStateException` comme exception domaine
**Constat :** `DraftSimultaneousService` lève `IllegalStateException` pour des violations domaine (fenêtre pas en OPEN, double soumission). Le handler `@ExceptionHandler(IllegalStateException.class)` est trop large — intercepte les ISE Spring/Hibernate.
**Action :** Règle projet : toute exception domaine doit être une sous-classe d'une exception projet (ex: `DraftWindowViolationException extends RuntimeException`). Ne jamais utiliser `IllegalStateException` ou `IllegalArgumentException` directement comme signal domaine — elles sont trop génériques.

#### Leçon L2 — Tests d'autorisation dans le template de story
**Constat :** 6 stories ont nécessité des tests de sécurité créés en code review. C'est le même finding répété 6 fois.
**Action :** Ajouter dans le template de story : "Task [N]: Tests d'autorisation Spring Security pour chaque controller. Inclure : unauthenticated → 401/403, non-admin → 403 si endpoint admin, authentifié → passe." Cette tâche est obligatoire pour toute story créant un controller.

#### Leçon L3 — `HashSet.contains()` vs `List.contains()` pour les lookups exclusivité
**Constat :** `List.contains()` sur la liste des picks existants est O(n). Pour un draft avec 100 picks, c'est 100 comparaisons par validation. Identifié deux fois (4.5 L1, 4.6 L2) sans être corrigé.
**Action :** Toute opération "is X in this collection?" sur une liste extraite de DB doit utiliser `new HashSet<>(list).contains(x)`. Ajouter dans `project-context.md` ce pattern standard.

#### Leçon L4 — Document le contexte "Already Done" pour les stories de validation
**Constat :** Stories 4.3 et 4.4 sont des stories BMAD dont 80% du scope était déjà implémenté dans JIRA-UX-003c. La section "Technical Context Already Done" est utile mais les tâches cochées à l'avance créent de la confusion.
**Action :** Pour les stories "validation/gap-filling", ajouter une section explicite "SCOPE RÉEL DE CETTE STORY : [liste précise des seules choses à faire]". Ne pas cocher les tâches pré-existantes comme "done" — les documenter en "Technical Context".

#### Leçon L5 — `Optional.orElse(null)` = code smell immédiat
**Constat :** `pickFirstConflict().orElse(null)` dans `DraftSimultaneousService` (4.4 M1) est un anti-pattern — null-propagation explicite alors que `Optional` est précisément là pour l'éviter.
**Action :** Règle : `Optional.orElse(null)` est interdit dans le codebase. Utiliser `orElseThrow()` ou un autre mécanisme. Linter (Spotless/Checkstyle) peut enforcer cette règle si ajouté.

### 4.4 — 5 Whys Deep Dive

#### Problème : Tests de sécurité manquants — 6ème occurrence consécutive

**Pourquoi 1 :** Les tests de sécurité sont systématiquement créés en code review, jamais en story.
↓
**Pourquoi 2 :** Les tâches de la story ne mentionnent pas les tests d'autorisation.
↓
**Pourquoi 3 :** Le template de story ne contient pas de section "Tests de sécurité".
↓
**Pourquoi 4 :** La DoD ne liste pas "Tests d'autorisation pour chaque controller endpoint" comme condition.
↓
**Cause racine :** Template de story et DoD incomplets sur la dimension sécurité.

**Action :** C'est la 6ème fois. La solution est connue : template de story + DoD avec section sécurité obligatoire. Cette action est prioritaire avant tout travail futur.

---

#### Problème : `IllegalStateException` utilisée comme exception domaine

**Pourquoi 1 :** `DraftSimultaneousService.requireOpenWindow()` lève `IllegalStateException("Window is not OPEN")`.
↓
**Pourquoi 2 :** `IllegalStateException` est une exception Java standard "prête à l'emploi" — pas besoin de créer une classe.
↓
**Pourquoi 3 :** Aucune règle dans le projet n'interdit l'utilisation d'exceptions Java standard comme signal domaine.
↓
**Cause racine :** Absence de règle sur les types d'exceptions domaine acceptables.

**Action :** Règle dans `project-context.md` : exceptions domaine = toujours une classe projet. `IllegalStateException`, `IllegalArgumentException`, `RuntimeException` directes = interdites comme signal métier.
