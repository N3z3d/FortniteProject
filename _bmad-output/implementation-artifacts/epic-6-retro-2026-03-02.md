## 3. Ce qui peut être amélioré

### Gaps identifiés

- **`@Transactional` self-invocation (6.2 L1) — dette technique réelle** — `computeDeltasForGame()` est annoté `@Transactional` mais appelé depuis `computeAllGameDeltas()` dans la même classe → Spring AOP bypass le proxy → la transaction n'est pas honorée. Risque bas (batch idempotent, quotidien), mais c'est une dette architecturale documentée non adressée.
- **Branche `playerRepository.findById().isEmpty()` non couverte (6.2 L2)** — La branche défensive "joueur introuvable → contribution 0" (AC #2) n'est pas couverte dans `TeamScoreDeltaBatchServiceTest`. Test de couverture manquant.
- **Security tests toujours en code review** — 6.1 (M1) et 6.3 (H1) nécessitent des security tests créés en review. 4ème epic consécutif avec ce même pattern. Action A2/A5 des rétros précédentes jamais implémentée.
- **`GameLeaderboardControllerTest` Mockito pur vs `@WebMvcTest`** — Le test controller utilise `@ExtendWith(MockitoExtension.class)` au lieu de `@WebMvcTest` → `@PreAuthorize("isAuthenticated()")` non exercé dans le test unitaire. Le security test dédié compense mais c'est une inconsistance de pattern.
- **ScoreCalculationService corrompu — jamais résolu** — Le fichier reste dans le codebase avec du double-encoded UTF-8. La solution "créer un nouveau service" est pragmatique mais laisse une bombe silencieuse. Une story de cleanup spécifique est nécessaire.
- **Route leaderboard `:id/leaderboard` migrée de `loadChildren` → `loadComponent`** — Cette migration est correcte pour Angular standalone, mais le module `LeaderboardModule` reste dans le codebase sans que la route principale `/leaderboard` ne soit migrée. Inconsistance architecturale frontend.

### 4.2 Lessons Learned Extraction — 5 Leçons

#### Leçon L1 — Leçons de rétro inter-epic : `@Scheduled` appliqué
**Constat :** Epic 1 avait identifié l'absence de `@Scheduled` sur `PlayerQualityService` comme gap critique. Epic 6 a appliqué le pattern "orchestrateur séparé" dès la story 6.2.
**Validation :** Le processus de rétro fonctionne quand les actions sont précises et mémorisées (MEMORY.md en l'occurrence). Cette leçon est la preuve que la rétro a de la valeur.

#### Leçon L2 — `@Transactional` sur méthode privée dans la même classe : toujours Spring AOP piège
**Constat :** `computeDeltasForGame()` est `@Transactional` mais appelé depuis la même classe → Spring AOP n'intercepte pas le call interne → la transaction ne démarre pas.
**Action :** Règle projet : `@Transactional` sur une méthode privée ou appelée depuis la même classe = inefficace. Déplacer la méthode dans un service séparé ou utiliser `@TransactionTemplate` injecté.

#### Leçon L3 — Workaround "nouveau service" vs correction
**Constat :** `ScoreCalculationService` corrompu → créer `TeamScoreDeltaBatchService` à côté. Solution pragmatique et correcte pour éviter la corruption. Mais le fichier corrompu reste.
**Action :** Tout workaround "fichier dangereux, ne pas toucher" doit être accompagné d'un ticket de cleanup dans le backlog. Sans ticket, la dette s'accumule silencieusement.

#### Leçon L4 — `@PreAuthorize` vs `userResolver` : choisir un pattern et s'y tenir
**Constat :** 6.3 introduit `@PreAuthorize("isAuthenticated()")` sur `GameLeaderboardController`. D'autres controllers (SwapSoloController, DraftParticipantTradeController) utilisent encore `userResolver.resolve()` + `if (null) return 401`. Deux patterns pour la même chose.
**Action :** Standardiser : `@PreAuthorize("isAuthenticated()")` pour les endpoints authentifiés (clean, déclaratif), `@PreAuthorize("hasRole('ADMIN')")` pour l'admin. Supprimer progressivement le pattern `userResolver` manuel.

#### Leçon L5 — Tests de couverture : les branches défensives sont les plus importantes
**Constat :** 6.2 L2 : la branche `playerRepository.findById().isEmpty() → contribution = 0` n'est pas testée. C'est précisément la branche "batch ne plante pas" de l'AC #2. Les branches défensives ont tendance à ne pas être testées parce qu'elles semblent "évidentes".
**Action :** En code review, vérifier spécifiquement les branches `if (x.isEmpty()) continue` — elles sont souvent non couvertes et critiques pour la robustesse en production.

### 4.4 — 5 Whys Deep Dive

#### Problème : Tests de sécurité manquants — 4ème epic consécutif

**Pourquoi 1 :** Security tests trouvés en code review pour la 4ème fois consécutive.
↓
**Pourquoi 2 :** Les tâches de story ne demandent pas explicitement des tests d'autorisation.
↓
**Pourquoi 3 :** L'action "modifier le template de story" est notée dans chaque rétro mais jamais exécutée.
↓
**Pourquoi 4 :** L'action n'a pas de responsable, pas de date, pas de ticket. C'est une intention.
↓
**Cause racine :** Les actions de rétro sans ticket et sans responsable assigné ne sont jamais exécutées. La rétro génère des intentions, pas des actions.

**Action :** Créer immédiatement un ticket "JIRA-PROC-001 : Modifier template de story — ajouter Task Sécurité obligatoire pour tout controller". Assigner à SM. Cette action ne doit plus être dans une rétro — elle doit être dans le backlog.
