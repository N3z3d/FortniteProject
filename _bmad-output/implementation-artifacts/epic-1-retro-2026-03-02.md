## 2. Ce qui a bien fonctionné

### Points techniques

- **Décision StubResolutionAdapter** : L'impossibilité d'obtenir des clés API Fortnite Tracker dès le sprint 1 aurait pu tout bloquer. La décision d'injecter un `ResolutionPort` swappable avec `StubResolutionAdapter` par défaut est la bonne réponse architecturale — conforme DIP. Le système peut fonctionner en closed-loop et sera connecté à la vraie API quand FEAT-003/FEAT-004 seront débloqués.
- **Pattern non-bloquant établi** : Les stories 1.3 et 1.4 ont formalisé le pattern "try-catch + log WARN + continue" pour les opérations d'audit non-critiques (persist identity, persist alias). Ce pattern a été réutilisé systématiquement dans toutes les stories suivantes qui touchent à des opérations annexes.
- **Architecture hexagonale tenue** : `PlayerIdentityEntry`, `PlayerAliasEntry` sont des domaines `final`, indépendants de Spring/Hibernate. Les ports (`PlayerIdentityRepositoryPort`, `PlayerAliasRepositoryPort`, `PrSnapshotQueryPort`) respectent l'inversion de dépendance. `DomainIsolationTest` vert sur toutes les stories.
- **PlayerQualityService autonome** : Story 1.5 a livré une architecture propre — 3 dépendances seulement, clairement séparée de `PrIngestionRowProcessor`. Séparation des responsabilités ingestion (1.1–1.4) vs qualité batch (1.5).
- **Append-only persistence** : Le pattern snapshots PR + alias append-only (1.4) est robuste — pas d'upsert risqué, chaque ingestion crée un log audit immuable. Composite key `(player, region, date)` garantit l'unicité sans logique métier complexe.
- **Fondation d'Epic 7 solide** : `PlayerIdentityEntry` + `PlayerIdentityRepositoryPort` créés en 1.3 sont devenus la fondation complète d'Epic 7 (supervision admin pipeline). Le modèle domaine était suffisamment solide pour supporter 7+ epics d'extension sans modification des fondations.

### Points process

- **Progression incrémentale disciplinée** : Chaque story a ajouté exactement une responsabilité à `PrIngestionRowProcessor` (1 dépendance par story, 1.1 → 1.4). Pas de big bang — chaque incrément était sûr, testable, réversible.
- **Stories bien découpées** : Les 5 stories couvrent des responsabilités disjointes : parsing CSV (1.1), résolution (1.2), audit UNRESOLVED (1.3), audit alias (1.4), qualité batch (1.5). Zéro overlap de scope.
- **Zéro régression** : Le système de tests TDD a permis de valider chaque ajout sans casser l'existant, malgré 4 mises à jour consécutives de `PrIngestionRowProcessor`.

## 3. Ce qui peut être amélioré

### Gaps identifiés

- **Pas de code review adversariale** : Epic 1 n'a pas bénéficié du workflow `bmad-bmm-code-review`. Les patterns de resilience identifiés en Epic 2 (try-catch warmup, tests Spring AOP) auraient pu être introduits dès Epic 1. C'est le seul epic sans review — et c'est justement l'epic fondateur.
- **PrIngestionRowProcessor : constructeur à 5 args** : La croissance 3 → 5 dépendances sur 4 stories consécutives a créé une cascade de mises à jour de tests. Chaque story obligeait la mise à jour de 3 fichiers (`PrIngestionServiceTddTest`, `PrIngestionServiceRuntimePortsTest`, `PrIngestionRowProcessorUnresolvedTest`). Un `IngestionAuditFacade(identityPort, aliasPort)` extrait dès story 1.3 aurait regroupé les 2 dépendances d'audit.
- **StubResolutionAdapter toujours actif en production** : Tous les `PlayerIdentityEntry` créés ont un `epicId` synthétique ou null — la data produite est inutilisable pour l'analytics réelle tant que FEAT-003/004 n'est pas livré. Ce fait critique n'est documenté explicitement dans aucune user story.
- **`PrIngestionServiceTddTest` : @Import croissant** : Le test `@DataJpaTest` accumule des `@Import(A.class, B.class, C.class...)` à chaque story. Ce pattern crée de la dette test — tout changement de dépendance casse ce fichier. Un row processor ne devrait pas être testé dans un contexte `@DataJpaTest`.
- **`@Scheduled` absent sur `PlayerQualityService`** : Le job qualité `runDailyQualityJob()` n'a pas d'annotation `@Scheduled` — il ne se déclenche jamais automatiquement en production. Les alertes UNRESOLVED >24h et détections de doublons epicId sont implémentées mais inactives. Ce gap opérationnel n'a été identifié qu'en rétrospective.

## 4. Élicitation Avancée — 5 Méthodes

### 4.1 Hindsight Reflection — Vue depuis mars 2026 (7 epics plus tard)

**Pattern 1 — La décision StubResolutionAdapter a traversé tout le sprint**
Le `StubResolutionAdapter` a été utilisé dans les epics 1 à 7 sans jamais être remplacé. 7 epics plus tard, le projet n'a toujours pas de vraie connexion Fortnite API — et le système fonctionne. C'est la définition d'une bonne décision architecturale sous contrainte : les fonctionnalités (catalogue Epic 2, draft Epics 3–5, leaderboard Epic 6, supervision Epic 7) fonctionnent toutes avec des Epic IDs synthétiques. La dette est bornée et clairement localisée dans FEAT-003/004.

**Pattern 2 — `PrIngestionRowProcessor` à 5 deps est resté le plafond**
Après Epic 1, le constructeur avait 5 arguments. Aucune story suivante n'en a ajouté un 6ème. Le risque d'explosion de complexité ne s'est pas matérialisé. Cependant, la cascade de mises à jour de tests sur 4 stories aurait pu être évitée avec un refactoring préventif dès story 1.3.

**Pattern 3 — `PlayerAliasEntry` + `PlayerIdentityEntry` : fondation d'Epic 7**
Ces deux modèles domaine créés en Epics 1.3/1.4 ont directement supporté les stories 7.1 (dashboard pipeline) et 7.2 (correction manuelle de fiche joueur). `PlayerIdentityEntry.correctMetadata()` en 7.2 a été ajouté sans modifier les fondations. Le modèle était suffisamment ouvert pour l'extension — OCP respecté.

**Pattern 4 — Le scheduler non configuré est resté silencieux**
`PlayerQualityService.runDailyQualityJob()` n'a jamais été déclenché automatiquement. En production réelle, les alertes de doublons epicId et les UNRESOLVED >24h ne seraient jamais émises. Ce bug opérationnel a traversé 7 epics sans être détecté parce qu'aucun test ne vérifie l'activation.

### 4.2 Lessons Learned Extraction — 5 Leçons

#### Leçon L1 — Documenter l'impact des stubs en production
**Constat :** `StubResolutionAdapter` est une décision architecturale valide. Mais aucune story ne documente "les données produites sont inutilisables pour l'analytics réelle tant que FEAT-003 n'est pas livré". Ce fait est implicite.
**Action :** Pour toute story impliquant un stub/mock en production, ajouter une note explicite dans les Dev Notes : "Stub actif — impact sur qualité des données : [description précise]".

#### Leçon L2 — Extraire une facade d'audit avant que les dépendances prolifèrent
**Constat :** En 4 stories consécutives, `PrIngestionRowProcessor` a reçu 4 nouvelles dépendances, obligeant la mise à jour de 3 fichiers de test à chaque story. Le seuil de refactoring aurait dû être 2 dépendances "audit/observabilité".
**Action :** Quand 2+ dépendances de même nature (audit, persistance annexe) s'accumulent dans un service, extraire un `XxxFacade` dès la 2ème. Règle : ≤ 4 dépendances dans un row processor.

#### Leçon L3 — Tester l'activation du scheduler, pas seulement la logique
**Constat :** `PlayerQualityService.runDailyQualityJob()` a été testé pour sa logique interne, mais pas pour son activation automatique. Le `@Scheduled` manquant n'a pas été détecté.
**Action :** Pour tout service batch/scheduler, ajouter un test d'activation : vérifier que l'annotation `@Scheduled` est présente sur la méthode de déclenchement. Alternativement, tester le déclenchement dans un context Spring (`@SpringBootTest`).

#### Leçon L4 — Séparer les tests unitaires des tests @DataJpaTest pour les row processors
**Constat :** `PrIngestionServiceTddTest` (test `@DataJpaTest`) a accumulé des `@Import` à chaque story. Ce pattern mélange test de logique métier (unitaire, Mockito) et test de persistance (slice, Spring Data). Le résultat est un test fragile et difficile à maintenir.
**Action :** Row processor : tester la logique métier avec Mockito uniquement (toutes les dépendances mockées). Tester les requêtes repository séparément dans un `@DataJpaTest` dédié uniquement au repository.

#### Leçon L5 — Code review adversariale obligatoire dès la première story
**Constat :** Epic 1 est le seul epic sans code review. Les issues de design (constructor 5 args, @DataJpaTest fragile, scheduler manquant) n'ont été identifiées qu'en rétrospective. Ces patterns se sont propagés sur 4 stories sans feedback adversarial.
**Action :** Le workflow `bmad-bmm-code-review` est obligatoire à partir de la 1ère story d'un epic. La review sur la story fondatrice (1.1 pour le pipeline) est la plus critique — c'est là que les patterns se fixent pour toutes les stories suivantes.

### 4.4 — 5 Whys Deep Dive

#### Problème : `PlayerQualityService` ne se déclenche jamais automatiquement

**Pourquoi 1 :** Pas d'annotation `@Scheduled` sur `runDailyQualityJob()`.
↓
**Pourquoi 2 :** La story 1.5 définissait "job qualité" mais ne spécifiait pas de planification cron dans les Acceptance Criteria.
↓
**Pourquoi 3 :** Les AC se concentraient sur le résultat retourné (`PlayerQualityJobResult`) et non sur l'activation en production.
↓
**Pourquoi 4 :** La story a été créée sans répondre à "comment ce job est-il déclenché en production ?"
↓
**Pourquoi 5 (root cause) :** Le template de story ne contient pas de champ "Déclenchement prod" pour les composants batch/scheduler.

**Action :** Pour toute story créant un service batch, ajouter dans le template : "Déclenchement prod : [scheduler cron / event bus / manuel / test only]". Si `@Scheduled`, l'expression cron doit être dans les Dev Notes.

---

#### Problème : Code review absente sur Epic 1

**Pourquoi 1 :** `bmad-bmm-code-review` n'a pas été exécuté.
↓
**Pourquoi 2 :** Le workflow code review n'était peut-être pas encore configuré au moment d'Epic 1.
↓
**Pourquoi 3 :** La Definition of Done ne mentionnait pas le code review comme étape obligatoire.
↓
**Cause racine :** DoD non formalisée — le même constat que dans la rétro Epic 2. Les deux épics fondateurs (1 et 2) ont tous deux identifié l'absence de DoD comme cause racine. Cela confirme que la DoD formalisée dans `project-context.md` est une action critique.

**Action :** Ajouter une section DoD dans `project-context.md` (aligné avec la recommandation L1 de la rétro Epic 2).

## 8. Mot de clôture

Epic 1 a posé les fondations du pipeline d'ingestion. La décision la plus importante — `StubResolutionAdapter` comme port swappable — a traversé l'intégralité du sprint (7 epics) sans jamais bloquer une seule feature. C'est la définition d'une bonne décision architecturale sous contrainte.

Le seul regret notable : l'absence de code review adversariale sur cet epic fondateur. Les issues identifiées en rétrospective (constructor 5 args, scheduler non configuré, @DataJpaTest fragile) auraient probablement été corrigées dès story 1.1 avec une review. Les patterns qui se fixent sur l'epic fondateur se propagent sur tous les epics suivants.

**Action principale :** Configurer le `@Scheduled` manquant sur `PlayerQualityService` avant la mise en production réelle.
