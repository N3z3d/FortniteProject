## 2. Ce qui a bien fonctionné

### Points techniques

- **`AdminGameSupervisionService` : 1 seule dépendance** — Service de supervision avec 1 dep (`GameRepository`). C'est le service le plus simple du sprint et un modèle de SRP : une responsabilité unique (assembler les données de supervision), une seule source de données.
- **7.4 code review : 2H + 2M + 2L fixés** — La densité de fixes la plus élevée du sprint. En particulier H2 (validation statut invalide → 400 dans le service) est une correction défensive critique : sans elle, `FINISHED` ou `CANCELLED` auraient retourné une liste vide (comportement trompeur) au lieu de 400 (comportement correct).
- **`RegionalStatRow` domain record** — Nouveau pattern : domain record pour les résultats de requêtes agrégées `(region, status, count)`. Propre, testable, conforme au `DomainIsolationTest`. Pattern réutilisable pour futures queries agrégées.
- **`CorrectMetadataDialogComponent` standalone** — Dialog avec `MAT_DIALOG_DATA` typé (`CorrectionDialogData`/`CorrectionDialogResult`), pré-rempli depuis `correctedUsername ?? playerUsername`. Pattern bien établi pour les futurs dialogs d'administration.
- **`Clock` double-constructor — pattern réutilisé** — `UnresolvedAlertService` utilise le même pattern double-constructeur pour les tests déterministes que `VisitTrackingService`. Cohérence des patterns d'infrastructure temporelle dans le projet.
- **Filtrage local dans 7.4** — `AdminGamesSupervisionComponent` charge toutes les parties en 1 requête HTTP et filtre localement via `computed()`. Pas de nouveau appel HTTP par onglet. Optimisation UX et performance correctement choisie.
- **`getRegionalStats()` : 2 queries au lieu de N** — `PlayerIdentityPipelineService.getRegionalStats()` utilise 2 queries JPQL (`countByRegionAndStatus` + `findLastIngestedAtByRegion`) agrégées en Java. Évite N requêtes pour N régions.

### Points process

- **7.1 : 0H/0M en code review** — Deuxième story du sprint (après 6.2) sans finding majeur. Pattern : stories d'extension (ajout de méthodes sur services existants) ont tendance à avoir moins de risques.
- **7.2 : Diagnostic M1 précis** — Le finding M1 (dialog pre-fill depuis `null` au lieu de `playerUsername`) a été trouvé et corrigé proprement, avec le test mis à jour ("initializes form with player username when no existing correction" au lieu de "initializes form with null").

## 3. Ce qui peut être amélioré

### Gaps identifiés

- **Story 7.3 : code review non documentée** — `7-3-alerting-escalant-et-logs-scraping-detailles.md` a `Status: review` mais `sprint-status.yaml` indique `done`. La section `Review Follow-ups` est absente du fichier story. Soit la review n'a pas été effectuée, soit elle n'a pas été documentée. C'est un gap de traçabilité : on ne sait pas si des findings ont été trouvés et corrigés.
- **`AdminPipelinePageComponent` — accumulation de complexité** — La page admin pipeline a maintenant 4 onglets (en attente / traités / par région / logs scraping), 5 observables dans le `forkJoin`, + chargement séparé pour les stats régionales, + `pipelineAlert`. Le composant risque de dépasser 500 lignes. Une extraction en sous-composants est à prévoir.
- **7.4 H2 : validation service absente de la story initiale** — `getActiveGamesByStatus()` n'avait pas de validation sur les statuts non-actifs. L'AC #3 ("une valeur invalide retourne 400") était explicite. La validation aurait dû être dans le code initial, pas découverte en code review.
- **`toDto()` et `toResponse()` dispersés** — `AdminGameSupervisionService.toDto()`, `PlayerIdentityPipelineService.toResponse()`, `TeamDeltaLeaderboardService.buildUsernameMap()` — pattern "assembler des données depuis plusieurs sources" répété sans abstraction commune. Acceptable pour l'instant mais source de duplication croissante.
- **Régions codées en dur** — `CorrectMetadataDialogComponent` définit `REGIONS = ['EU', 'NAW', 'BR', 'ASIA', 'OCE', 'NAC', 'ME', 'NA', 'UNKNOWN']` localement. Devrait utiliser l'enum `PlayerRegion` de l'API (ou un endpoint dédié) pour éviter la désynchronisation future.

## 4. Élicitation Avancée — 5 Méthodes

### 4.1 Hindsight Reflection — Vue depuis la fin du sprint

**Pattern 1 — Epic 7 comme epic de consolidation et supervision**
Les 4 stories d'Epic 7 sont toutes des features d'administration : dashboard régional, correction manuelle, alerting, supervision globale. Aucune n'étend le domaine métier (pas de nouvelle règle draft, pas de nouveau scoring). C'est normal pour un epic de fin de sprint — l'opérateur a besoin de visibilité avant la mise en production.

**Pattern 2 — `PlayerIdentityEntry.restore()` à 16 paramètres**
Story 7.2 a étendu `restore()` de 12 à 16 paramètres (`@SuppressWarnings("java:S107")`). C'est un signal architectural : `PlayerIdentityEntry` accumule trop de responsabilités. Un futur refactoring devrait extraire les champs de correction dans un `ValueObject` dédié (`MetadataCorrection`).

**Pattern 3 — AdminPipelinePageComponent : composant en surpoids**
Sur 4 stories (7.1, 7.2, 7.3 et une partie d'Epic précédent), le même composant a été étendu. Il gère maintenant les entrées non résolues, résolues, les stats régionales, les logs de scraping, et les alertes pipeline. Un seul composant avec 5+ responsabilités. Une refactorisation en sous-composants est nécessaire avant qu'il dépasse 500 lignes.

### 4.2 Lessons Learned Extraction — 5 Leçons

#### Leçon L1 — AC #3 non vérifié → H2 en code review
**Constat :** L'AC #3 de 7.4 stipule explicitement "une valeur invalide retourne 400". `getActiveGamesByStatus()` acceptait `FINISHED`/`CANCELLED` sans erreur → comportement trompeur. Le dev n'a pas vérifié tous les ACs avant de soumettre.
**Action :** Règle de DoD : avant de soumettre une story pour review, lire chaque AC ligne par ligne et vérifier qu'il est couvert par un test ou une validation de code. Un AC non couvert = story incomplete.

#### Leçon L2 — Story 7.3 sans review documentée
**Constat :** Le sprint-status indique `7-3-alerting-escalant-et-logs-scraping-detailles: done` mais le fichier story a `Status: review`. La section `Review Follow-ups` est absente. On ne sait pas si des findings ont été trouvés.
**Action :** Règle : une story ne passe à `done` dans le sprint-status que si la section `Review Follow-ups` est présente dans le fichier story (même si vide — "Aucun finding"). La revue est une étape du DoD, pas optionnelle.

#### Leçon L3 — `PlayerIdentityEntry` à 16 params : extraction de ValueObject
**Constat :** `restore()` a 16 paramètres. `@SuppressWarnings("java:S107")` masque le problème mais ne le résout pas. Les 4 champs de correction (`correctedUsername`, `correctedRegion`, `correctedBy`, `correctedAt`) sont un ValueObject naturel.
**Action :** Lors du prochain refactoring de `PlayerIdentityEntry`, extraire `MetadataCorrection(username, region, by, at)`. Réduire `restore()` à 13 paramètres.

#### Leçon L4 — Régions codées en dur : source de désynchronisation
**Constat :** `CorrectMetadataDialogComponent` définit la liste des régions localement. Si de nouvelles régions sont ajoutées à l'enum `PlayerRegion`, le frontend doit être mis à jour manuellement.
**Action :** Créer un endpoint `GET /api/meta/regions` ou exposer l'enum via un endpoint existant. Le frontend récupère dynamiquement les régions valides.

#### Leçon L5 — `Clock` pattern : standardisé et réutilisé correctement
**Constat :** `UnresolvedAlertService` utilise le même double-constructor Clock pattern que `VisitTrackingService`. C'est exactement ce que la mémorisation de patterns inter-epics permet. Le pattern est documenté dans MEMORY.md et appliqué sans hésitation.
**Action :** Documenter les patterns "copiés/réutilisés" dans `project-context.md` pour qu'ils soient disponibles à tout nouveau dev sans avoir à lire MEMORY.md.

### 4.4 — 5 Whys Deep Dive

#### Problème : Story 7.3 en `Status: review` dans le fichier mais `done` dans sprint-status

**Pourquoi 1 :** Le sprint-status a été mis à jour à `done` sans que le fichier story ne soit mis à jour.
↓
**Pourquoi 2 :** Le SM/Dev a marqué la story comme `done` sans vérifier que le fichier story reflète le statut réel.
↓
**Pourquoi 3 :** Il n'y a pas de vérification croisée entre sprint-status et les fichiers story lors du passage à `done`.
↓
**Pourquoi 4 :** Le workflow de clôture de story n'inclut pas de checklist "vérifier que la section Review Follow-ups est présente".
↓
**Cause racine :** Le DoD ne définit pas explicitement ce qui doit être présent dans le fichier story pour passer à `done`. La review est traitée comme facultative.

**Action :** Ajouter dans le DoD : "La section `Review Follow-ups` doit être présente dans le fichier story (même vide) avant que le sprint-status ne passe à `done`."

Ce sprint de 7 epics (36 stories) a couvert l'intégralité du périmètre prévu : pipeline d'ingestion, catalogue joueur, gestion de partie, draft (serpent + simultané), échanges, scoring et supervision admin. Le produit est fonctionnellement complet.

3. **Architecture hexagonale tient à l'échelle** — 36 stories, 0 violation `DependencyInversionTest`, 0 violation `DomainIsolationTest`. Les tests d'architecture automatiques (ArchUnit) ont joué leur rôle défensif tout au long du sprint.

**3 dettes techniques prioritaires pour le prochain sprint :**
1. **Story 7.3 review** — À compléter immédiatement.
2. **`AdminPipelinePageComponent` refactorisation** — Avant dépassement 500 lignes.
3. **Template de story avec Task Sécurité obligatoire** — Action répétée 4+ fois sans ticket concret. Cette fois : `JIRA-PROC-001` avant toute autre chose.
