## 3. Ce qui peut être amélioré

### Gaps identifiés
- **Task checkboxes incomplètes en fin de story** : Sur 2.4, la task "appelle `findByRegion(region)` pour chaque valeur de `PlayerRegion`" était cochée mais incluait UNKNOWN à tort. Les task checkboxes ont été remplies avant que le code review ne détecte les issues.
- **Tests de cache annotation-only** : Strategy "réflexion d'annotation" trop faible — ne prouve pas que Spring AOP intercepte réellement. A noter dans le template de story pour la prochaine utilisation de `@Cacheable`.
- **Pas de Definition of Done formalisée** : Le project n'a pas de DoD écrite. On s'est appuyé sur "tasks cochées + Completion Notes + File List" — bon, mais implicite.
- **project-context.md statique** : Le fichier n'a pas été enrichi après Epic 1. Les patterns découverts en Epic 2 (Spring AOP integration test, warmup pattern, `PlayerRegion.UNKNOWN` edge case) ne sont nulle part documentés formellement.

## 4. Élicitation Avancée — 5 Méthodes

### 4.1 Hindsight Reflection — Vue depuis septembre 2026

En regardant en arrière depuis 6 mois après la livraison :

**Pattern 1 — L'infrastructure cache réutilisable**
`CacheConfig` + `PlayerCatalogueWarmupService` sont devenus le template standard pour tous les caches ajoutés en Epics 4-6. Le pattern "warmup sur ApplicationReadyEvent + méthode publique callable par scheduler" a été copié 3 fois. Avoir la bonne abstraction dès Epic 2 a économisé du travail.

**Pattern 2 — La discipline test AOP**
L'intégration de `@ExtendWith(SpringExtension.class)` + `@ContextConfiguration` + `@EnableCaching` + `ConcurrentMapCacheManager` est devenue le standard de test pour toutes les méthodes `@Cacheable`. Sans le code review F-001 qui a forcé ce pattern, on aurait eu des faux positifs en test jusqu'en Epic 5.

**Pattern 3 — Exclusion des valeurs enum transitoires**
Le pattern `if (region == PlayerRegion.UNKNOWN) continue` dans les boucles sur `PlayerRegion.values()` a été appliqué systématiquement dans les stories 3.x et 4.x. Sans Epic 2, ce cas limité aurait été redécouvert en prod.

**Pattern 4 — Résilience au démarrage**
Le try-catch dans `warmup()` avec log WARN (pas ERROR) est devenu la convention pour tous les services `ApplicationListener`. Cela a évité un crash prod lors de la mise en production avec un Redis lent à démarrer.

### 4.2 Lessons Learned Extraction — 5 Leçons

#### Leçon L1 — Formaliser la Definition of Done
**Constat :** Les task checkboxes ont été cochées avant que le code review révèle des issues (UNKNOWN en warmup, pas de try-catch). La DoD implicite ne suffit pas.
**Action :** Ajouter dans `project-context.md` une section DoD formelle :
```
Definition of Done (story):
- [ ] Toutes les tasks cochées ET vérifiées après implémentation
- [ ] Completion Notes renseignées (ce qui a changé + pourquoi)
- [ ] File List exhaustive
- [ ] mvn spotless:apply && mvn test → 0 régression
- [ ] Code review approuvée (findings HIGH/MEDIUM fixés)
```

#### Leçon L2 — Tests Spring AOP : réflexion d'annotation insuffisante
**Constat :** Un test qui vérifie `method.getAnnotation(Cacheable.class) != null` ne prouve pas que le cache fonctionne. Une mauvaise config Spring (pas d'`@EnableCaching`, mauvais `CacheManager`) passe le test mais échoue en prod.
**Action :** Ajouter dans `project-context.md` le pattern `PlayerCatalogueServiceCacheIntegrationTest` comme template standard pour tout test de `@Cacheable`.

#### Leçon L3 — Template edge cases : valeurs enum transitoires
**Constat :** `PlayerRegion.UNKNOWN` est une valeur transitoire — elle ne doit jamais apparaître dans des boucles de traitement "tous les joueurs par région". Ce cas n'était pas documenté.
**Action :** Toute boucle sur `values()` d'un enum qui contient des états transitoires (`UNKNOWN`, `PENDING`, `DRAFT`) doit explicitement les exclure avec un `continue` commenté.

#### Leçon L4 — Code review adversariale : les MEDIUM sont les vrais risques
**Constat :** Les findings MEDIUM (F-002 try-catch, F-003 UNKNOWN) semblaient mineurs mais auraient causé des incidents en prod (crash startup si DB lente, warmup inutile). Les HIGH sont évidents — les MEDIUM demandent un regard adversarial.
**Action :** Maintenir le workflow code review adversariale obligatoire après chaque story. Le reviewer doit se demander "qu'est-ce qui se passe si l'infrastructure externe est indisponible ?"

#### Leçon L5 — Dépendances natives : Spring AOP vs self-invocation
**Constat :** En 2.4, la note de vigilance sur le self-invocation Spring AOP a évité un bug potentiel. Si `searchByNickname()` avait appelé `this.findAll()`, le cache ne se serait pas déclenché.
**Action :** Pour toute méthode `@Cacheable`, documenter dans la story : "appelée depuis [externe/interne] — [AOP OK / risque self-invocation]".

### 4.3 Pre-mortem Analysis — Risques Epic 3

#### Risque R1 — Ambiguïté Game domain vs legacy model (CRITIQUE)
**Scénario :** L'équipe implémente Story 3.1 (création de partie) en utilisant `model.Game` (JPA legacy) au lieu de `domain.game.model.Game` (hexagonal pur). La migration brownfield crée deux chemins de code parallèles — `GameCreationService` dépend de l'un, `GameQueryService` de l'autre. Les tests passent mais la cohérence architecturale est brisée.
**Prévention :** Décision explicite avant Story 3.1 : nouveau code = toujours `domain.game.model.Game`. Ajouter règle dans `project-context.md`. Vérifier dans code review que `model.Game` (JPA) n'apparaît jamais dans les nouvelles stories.

#### Risque R2 — Tests non-déterministes sur la création de partie (HAUT)
**Scénario :** Story 3.1 génère des codes d'invitation (UUID ou alphanumérique). Si les tests ne mockent pas le générateur, ils dépendent du random et échouent de façon intermittente en CI.
**Prévention :** Tout service générant des IDs/codes doit recevoir un `Supplier<String>` ou équivalent mockable. Prévoir dans la task "générateur injectable".

#### Risque R3 — Scope gap sur 3.3 (suppression de compte + libération roster)
**Scénario :** Story 3.3 "suppression de compte avec libération roster" touche à la fois le domaine User, le domaine Game, et potentiellement le domaine Trade. Sans liste exhaustive des entités à cascader, la story est incomplète et génère des données orphelines en prod.
**Prévention :** Story 3.3 doit lister toutes les entités cascadées dans les Acceptance Criteria avant le début de l'implémentation. Ajouter un test d'intégration vérifiant l'absence de données orphelines après suppression.

#### Risque R4 — Duplication de handlers d'exceptions (MOYEN)
**Scénario :** Epic 3 introduit de nouvelles exceptions métier (`GameNotFoundException`, `InvitationExpiredException`). Si chaque dev les ajoute dans `GlobalExceptionHandler` sans ordre, on revient à une classe de 525+ lignes (violant la limite 500L).
**Prévention :** Rappel en start-of-story : nouvelles exceptions métier → `DomainExceptionHandler` (@Order(1)), pas `GlobalExceptionHandler`. Vérifier en code review.

### 4.4 — 5 Whys Deep Dive

#### Problème observé : les task checkboxes étaient cochées mais incorrectes (UNKNOWN warmé)

**Pourquoi 1 :** La task disait "appelle `findByRegion(region)` pour chaque valeur de `PlayerRegion`" — cette formulation inclut UNKNOWN.
↓
**Pourquoi 2 :** La story a été créée avec le code de référence du Dev Notes incluant `PlayerRegion.values()` sans filtrage.
↓
**Pourquoi 3 :** Le Dev Notes reproduisait la doc produit (epics.md) qui ne mentionnait pas UNKNOWN comme cas spécial.
↓
**Pourquoi 4 :** L'enum `PlayerRegion.UNKNOWN` n'est pas documenté comme "valeur transitoire à exclure des traitements prod" dans `project-context.md`.
↓
**Pourquoi 5 (root cause) :** `project-context.md` est un artefact statique généré une fois, jamais mis à jour après chaque epic. Les patterns et conventions découverts pendant l'implémentation ne remontent pas.

**Cause racine 1 :** `project-context.md` n'est pas enrichi de façon incrémentale.
**Action :** Après chaque epic, le SM enrichit `project-context.md` avec les nouveaux patterns (section "Patterns confirmés").

---

**Problème observé : pas de Definition of Done formalisée**

**Pourquoi 1 :** Les stories ont une section "Tasks" avec checkboxes mais pas de DoD explicite.
↓
**Pourquoi 2 :** Le template de story ne contient pas de section DoD.
↓
**Pourquoi 3 :** BMAD v6 laisse la DoD à la discrétion de l'équipe dans `project-context.md`.
↓
**Cause racine 2 :** La DoD n'a jamais été formalisée dans le projet.
**Action :** Ajouter section DoD dans `project-context.md` (voir Leçon L1).
