## 4. Élicitation Avancée — 5 Méthodes

### 4.1 Hindsight Reflection — Vue depuis mars 2026 (2 epics plus tard)

**Pattern 1 — `requireTradePending()` comme modèle de guard domaine**
En Epics 6 et 7, aucune `IllegalStateException` non capturée n'est apparue. Le pattern "créer une exception domaine spécifique pour les violations d'état" (préconisé en Epic 4, appliqué ici) est maintenant ancré. La leçon a voyagé correctement d'une rétro à l'implémentation.

**Pattern 2 — `DraftAuditController` : toujours non consommé après 2 epics**
Epics 6 et 7 n'ont pas créé de page frontend pour le journal d'audit. L'endpoint backend est implémenté mais invisible pour les utilisateurs. C'est une feature "dark" : testée, fonctionnelle, mais sans valeur produit visible.

**Pattern 3 — SwapSoloService MAX 7 deps : signal architectural non traité**
Atteindre la limite exactement en 5.3 est un signal fort que ce service fait trop de choses. En Epics 6 et 7, le service n'a pas été touché. La refactorisation via `@TransactionalEventListener` pour découpler le swap de l'audit reste une dette.

### 4.2 Lessons Learned Extraction — 5 Leçons

#### Leçon L1 — "Créé en avance" = dette de scope si non testé
**Constat :** Story 5.1 a créé 8 fichiers pour 5.3 (fondations d'audit) hors scope défini. Ces fichiers ne sont pas testés dans 5.1 — les tests arrivent en 5.3. Si la spec de 5.3 avait changé, du code aurait été écrit inutilement.
**Action :** Si du code est créé en avance pour une future story, l'ajouter explicitement dans la section "Technical Context Already Done" de la story cible. Ne pas laisser ces fichiers dans un état "créés mais non testés dans cette story".

#### Leçon L2 — `requireTradePending()` : modèle de guard pré-condition domaine
**Constat :** `acceptTrade()`/`rejectTrade()` levaient `IllegalStateException` sur trade non-PENDING → interceptée comme `DRAFT_WINDOW_VIOLATION` (409, sémantiquement faux). La solution : `requireTradePending()` lève `InvalidSwapException` → 400 `INVALID_SWAP` (correct).
**Action :** Règle projet : toute méthode "d'action domaine" doit vérifier son pré-état avec une exception domaine spécifique avant d'exécuter. Pattern : `requireXxx()` → exception domaine avec code métier précis.

#### Leçon L3 — Feature backend sans UI = feature invisible
**Constat :** `DraftAuditController` (`GET /api/games/{gameId}/draft/audit`) existe depuis 5.3. 2 epics plus tard, aucune UI ne l'utilise.
**Action :** Lors de la création d'une story backend read-only, valider explicitement : "Y a-t-il une story frontend correspondante ?" Si non, documenter dans le backlog ou accepter explicitement que c'est une feature future.

#### Leçon L4 — O(n) `List.contains()` : 5 occurrences, 0 fix
**Constat :** 5 occurrences de `List.contains()` sur des UUIDs (4.5, 4.6, 5.1) notées en review LOW, jamais corrigées dans la même story.
**Action :** Les findings LOW "5 minutes ou moins" (comme `new HashSet<>(list).contains(id)`) doivent être corrigés dans la même story. Les LOW "plus longs" → créer une story dédiée immédiatement (pas "à la prochaine occasion").

#### Leçon L5 — `@DisplayName` : convention non enforced, 3 fois consécutives
**Constat :** Même finding LOW sur 3 stories de suite. Cela ne se résout pas en le notant à nouveau.
**Action :** Ajouter dans le template de story : "Tout bloc `@Nested` dans un test controller DOIT avoir `@DisplayName`. Vérifier avant de soumettre." Alternative : règle Checkstyle custom.

### 4.4 — 5 Whys Deep Dive

#### Problème : O(n) `List.contains()` identifié 5 fois, jamais corrigé

**Pourquoi 1 :** Le finding est noté LOW et laissé comme action item.
↓
**Pourquoi 2 :** Les action items LOW ne sont jamais traités dans la même story.
↓
**Pourquoi 3 :** Il n'y a pas de "phase de correction des LOWs" dans le workflow de story.
↓
**Pourquoi 4 :** L'équipe croit que les LOWs seront traités dans une story dédiée de dette technique.
↓
**Cause racine :** La story "dette technique LOW" est toujours déprioritée. Les LOWs simples (1-5 min) ne devraient jamais sortir de la story courante.

**Action :** Règle : tout LOW de type "1 ligne de code" (comme `new HashSet<>()`) est corrigé dans la même story. Documenter dans les directives de code review.
