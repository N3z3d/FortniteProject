## 2. Ce qui a bien fonctionné

### Points techniques

- **GAP CRITIQUE trouvé avant production** : Story 3.2 a identifié que `GameParticipantService.findGameFromRequest()` n'avait jamais validé l'expiration d'un code d'invitation — un bug de sécurité présent depuis l'implémentation initiale. L'analyse approfondie lors de la création de story a permis de le détecter. Fix : 4 lignes. Impact : sans ce fix, un code expiré aurait permis de rejoindre une partie indéfiniment.
- **Décision architecture Epic 2 → honorée** : La "règle absolue" formulée en Epic 2 retro (R1 — `domain.game.model.Game` exclusivement, jamais `model.Game` dans les services) a été explicitement documentée dans les Dev Notes de 3.1 et respectée sur toutes les stories.
- **Soft-delete User cohérent** : Story 3.3 a appliqué exactement le même pattern que le soft-delete `Game` (V30 migration). Cohérence totale : colonne `deleted_at`, `@Modifying @Query`, pattern port.
- **In-memory circular buffer** : Story 3.4 a réutilisé le pattern `ErrorJournalService` pour `GameIncidentService` — zéro migration DB requise, implémentation en ~50 lignes.
- **Architecture violations corrigées à la volée** : Story 3.4 a détecté 2 violations d'architecture pendant l'implémentation (`IncidentType` dans le mauvais package, `GameIncidentController` injectant directement un repository) et les a corrigées avant merge. `DependencyInversionTest` et `LayeredArchitectureTest` forcent ces corrections.
- **Code reviews efficaces** : 3 findings HIGH résolus avant merge — `IllegalStateException` capturée comme `DRAFT_WINDOW_VIOLATION` (sémantique fausse), `IncidentReportingService` sans tests, `LocalDateTime` au lieu d'`OffsetDateTime` pour les timestamps.

### Points process

- **Scope 3.3 bien défini** : L'action item A6 de la rétro Epic 2 ("lister exhaustivement les entités cascadées en AC") a été appliquée. Story 3.3 liste explicitement les statuts actifs (`CREATING`, `DRAFTING`, `ACTIVE` libérés) vs terminés (`FINISHED` conservés) dans les AC.
- **Exception handlers bien placés** : Toutes les nouvelles exceptions (`InvalidInvitationCodeException`, `AccountDeletionBlockedException`) ont été ajoutées dans `DomainExceptionHandler` (@Order(1)), pas dans `GlobalExceptionHandler`. L'action item R4 de la rétro Epic 2 a été respectée.
- **Stories bien découpées** : 3.1 (création) → 3.2 (invitations) → 3.3 (suppression compte) → 3.4 (incidents) forment une progression logique sans overlap.

## 3. Ce qui peut être amélioré

### Gaps identifiés

- **Story 3.1 Dev Record non rempli** : Les sections "Completion Notes" et "File List" de story 3.1 sont vides. Il est impossible de savoir quels fichiers ont été modifiés, combien de tests ont été ajoutés, ou quels problèmes ont été rencontrés. C'est une lacune de traçabilité critique pour l'epic fondateur des parties.
- **9 findings LOW en attente** : Les findings LOW sont acceptés mais s'accumulent : `serialVersionUID` manquant sur 2 exceptions, `clearAll()` public au lieu de package-private, `AdminIncidentListComponent` qui re-fetche HTTP à chaque filtre, `UserDeletionServiceTest` sans vérification d'ordre (`InOrder`), `leaveAllActiveGames()` qui charge tout en mémoire. Au-delà de 10 LOW en attente, ils commencent à former une dette visible.
- **Incidents in-memory = perte au redémarrage** : `GameIncidentService` (buffer circulaire 500 entrées) perd tous les incidents à chaque redémarrage. Story 3.4 le documente comme "acceptable pour le MVP" mais c'est une dette opérationnelle importante pour un système de supervision.
- **`UserDeletionServiceTest` sans vérification d'ordre** : La séquence `leaveGame()` avant `softDelete()` est critique pour la correction du comportement (si `softDelete()` est appelé en premier, le user est invalide quand `leaveGame()` tente de le retrouver). Le test ne vérifie pas cet ordre — c'est un test qui peut passer même avec une implémentation incorrecte.
- **Pas de code review documentée pour 3.1** : Story 3.1 a le plus grand impact structurel (enum `DraftMode`, enrichissement `Game` domaine, V36 migration) mais n'a pas de section "Review Follow-ups" visible dans le fichier story.

## 4. Élicitation Avancée — 5 Méthodes

### 4.1 Hindsight Reflection — Vue depuis mars 2026 (4 epics plus tard)

**Pattern 1 — `InvalidInvitationCodeException` : une sécurité qui aurait dû exister dès le début**
Le bug corrigé en 3.2 (validation expiration code) existait depuis l'implémentation initiale du système d'invitation. Si ce bug avait été en production, n'importe quel code expiré aurait permis de rejoindre une partie. Heureusement, la story spec l'a identifié comme "GAP CRITIQUE" avant l'implémentation. Cela rappelle que les specs de story sont elles-mêmes des outils de discovery de bugs.

**Pattern 2 — Le soft-delete User de 3.3 a tenu sur tous les epics suivants**
`User.deletedAt` introduit en 3.3 n'a pas eu besoin d'être modifié dans les epics 4–7. Le pattern "soft delete logique, PII conservées en base, filtre global optionnel" est robuste.

**Pattern 3 — `GameIncidentService` buffer circulaire : jamais migré en DB**
Sur les 4 epics suivants, `GameIncidentService` est resté in-memory. Les incidents sont perdus au redémarrage — mais cela n'a pas bloqué de feature. Bonne décision pour un MVP, mais à migrer avant prod réelle.

**Pattern 4 — Les violations d'architecture détectées à la compilation > détectées en revue**
Story 3.4 a eu ses violations (`IncidentType` mauvais package, repo direct dans controller) détectées par les tests d'architecture (`DependencyInversionTest`, `NamingConventionTest`) pendant l'implémentation. Le fait que ces tests existent et bloquent la compilation est plus efficace que n'importe quelle revue humaine.

**Pattern 5 — Story 3.1 sans traçabilité : un risque pour la maintenance**
L'absence de "File List" dans 3.1 signifie que si un bug apparaît dans la création de partie, il faut scanner manuellement les fichiers plutôt que se référer à la story. Ce pattern de non-documentation ne s'est pas répété sur 3.2–3.4.

### 4.2 Lessons Learned Extraction — 5 Leçons

#### Leçon L1 — Toujours remplir la section Dev Agent Record
**Constat :** Story 3.1 a son "File List" et "Completion Notes" vides. Cette section est la mémoire de l'implémentation — sans elle, il est impossible de faire un code review a posteriori ou de déboguer les régressions.
**Action :** Ajouter dans la DoD : "File List exhaustive dans la section Dev Agent Record". Le dev agent ne passe pas la story à `done` sans remplir cette section.

#### Leçon L2 — Valider l'ordre des opérations dans les services critiques
**Constat :** `UserDeletionService` doit appeler `leaveGame()` avant `softDelete()`. `UserDeletionServiceTest` ne vérifie pas cet ordre — un bug d'ordre passerait le test.
**Action :** Pour tout service dont l'ordre d'opérations est critique pour la correction, utiliser `InOrder` Mockito ou un test d'intégration qui vérifie l'état intermédiaire. Ajouter cette règle dans le template de test TDD.

#### Leçon L3 — Les findings LOW doivent être traités par batch
**Constat :** 9 findings LOW en attente après Epic 3. Ces findings sont "acceptés" individuellement mais s'accumulent. À 10+ LOW, la charge cognitive de les tracker dépasse le bénéfice de les reporter.
**Action :** À chaque rétro, traiter les LOW pendants si leur nombre dépasse 8. Consacrer 1 story "dette technique" par epic pour les LOW accumulés.

#### Leçon L4 — La spec de story est un outil de découverte de bugs
**Constat :** Le GAP CRITIQUE de 3.2 (validation expiration code) a été découvert pendant la création de la story spec, pas pendant un code review. Une analyse approfondie du code existant avant d'écrire les AC permet de trouver des bugs pre-existants.
**Action :** La phase "Dev Notes" d'une story doit toujours analyser le code existant et documenter les gaps ou bugs pré-existants. "Infrastructure déjà en place" est un pattern utile — mais il faut aussi lister "Infrastructure défectueuse / manquante".

#### Leçon L5 — Timestamps : toujours `OffsetDateTime` pour les données partagées
**Constat :** `IncidentEntry.timestamp` utilisait `LocalDateTime` (sans timezone) — corrigé en code review (M2) vers `OffsetDateTime`. Ce pattern s'était aussi posé en Epic 1 (snapshots PR). Les systèmes multi-utilisateurs ont besoin de timestamps timezone-aware.
**Action :** Règle projet : tout timestamp stocké ou sérialisé qui peut être vu par des utilisateurs en fuseaux différents doit être `OffsetDateTime` (pas `LocalDateTime`). Ajouter dans `project-context.md`.

### 4.4 — 5 Whys Deep Dive

#### Problème : Story 3.1 a son Dev Agent Record non rempli

**Pourquoi 1 :** Les sections "Completion Notes" et "File List" sont vides dans le fichier story.
↓
**Pourquoi 2 :** Le dev agent ne les a pas remplies après implémentation.
↓
**Pourquoi 3 :** La Definition of Done ne les exige pas formellement.
↓
**Pourquoi 4 :** La DoD est implicite dans ce projet (pas de section dédiée dans `project-context.md`).
↓
**Pourquoi 5 (root cause) :** Même cause racine identifiée en rétros Epics 1 et 2 — la DoD n'a pas été formalisée.

**Action :** La DoD doit inclure "File List exhaustive remplie" comme condition nécessaire pour passer une story à `done`. Ce point a été identifié 3 fois consécutives — il faut l'écrire dans `project-context.md` maintenant.

---

#### Problème : 9 findings LOW en attente après Epic 3

**Pourquoi 1 :** Les findings LOW sont acceptés individuellement en code review ("acceptable, LOW, pas bloquant").
↓
**Pourquoi 2 :** Il n'y a pas de règle sur le nombre maximum de LOW en attente avant qu'ils doivent être traités.
↓
**Pourquoi 3 :** Le processus de code review ne prévoit pas de mécanisme de flush de la dette technique.
↓
**Cause racine :** Absence de limite sur la dette LOW accumulée.

**Action :** Si le nombre de findings LOW ouverts dépasse 8, le SM crée une story "dette technique" dans le prochain sprint. Seuil : 8 LOW = 1 story dédiée.
